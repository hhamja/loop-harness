# 하네스 엔지니어링 정본 (Harness Engineering Canonical)

> **이 문서의 목적**: LLM 에이전트 하네스를 제로부터 설계·구현할 때의 기준이자,
> 기존 하네스의 코드·설계를 점검할 때의 판정 기준. 특정 프로젝트·도구에 독립적이며,
> 어떤 에이전트 런타임(Claude Code, Codex CLI, 자체 오케스트레이터)에도 적용된다.
>
> **이 문서를 읽는 에이전트에게**:
> - 모든 규칙에는 안정적인 ID가 있다 (AX-, PR-, AR-, IM-, AP-, CK-).
>   점검 결과를 보고할 때는 반드시 규칙 ID로 인용하라. 예: "AP-02 위반: verifier가 maker와 같은 세션에서 실행됨".
> - 충돌 시 우선순위: **공리(AX) > 원칙(PR) > 아키텍처(AR) > 구현(IM)**.
>   구현 디테일은 환경에 따라 변형 가능하지만, 공리를 깨는 변형은 결함이다.
> - 판정할 수 없는 항목은 추측으로 PASS/FAIL을 내리지 말고 "판정 불가 + 필요한 증거"로 보고하라.
> - Part 5의 체크리스트가 점검 작업의 실행 인터페이스다. Part 0–4는 그 체크리스트의 근거다.

## Part 0 — 공리 (Axioms)

하네스의 모든 설계 결정은 아래 12개 공리에서 도출된다. 공리끼리 충돌하는 상황이 오면
낮은 번호가 우선한다.

| ID | 공리 |
|----|------|
| AX-01 | **모델 출력은 신뢰 불가 입력이다.** 모델의 주장("테스트 통과했습니다", "파일 수정했습니다")은 검증되기 전까지 사실이 아니다. 하네스는 주장이 아니라 관측된 결과(exit code, 파일 상태, diff)로 판단한다. |
| AX-02 | **"done"은 기계가 판정한다.** 완료 조건은 명령이 0으로 종료 / 파일이 존재 / 출력이 일치하는 형태로 정의한다. 사람의 인상, 모델의 자기 평가, "그럴듯해 보임"은 완료 조건이 될 수 없다. |
| AX-03 | **만드는 자와 채점하는 자를 분리한다.** 산출물을 만든 컨텍스트가 그 산출물을 채점하면, 그것은 채점이 아니라 동의다. checker는 반드시 독립 컨텍스트에서 실행한다. |
| AX-04 | **상태는 디스크에 산다. 컨텍스트 윈도우는 캐시다.** 세션이 지금 죽어도, 새 세션이 디스크의 상태 파일만 읽고 작업을 이어갈 수 있어야 한다. 이 조건을 통과하지 못하는 상태는 상태가 아니다. |
| AX-05 | **자율 반경 = 검증기 신뢰도 × 되돌림 가능성.** 두 인수 중 하나라도 0에 가까우면 자율 실행 반경도 0에 가깝다. 자율성을 늘리고 싶으면 프롬프트가 아니라 검증기와 되돌림 수단에 투자한다. |
| AX-06 | **프롬프트는 요청이고, 훅은 강제다.** "절대 X를 하지 마라"는 지침은 확률적으로만 지켜진다. 반드시 지켜져야 하는 규칙은 결정론적 코드(훅, 게이트, 권한 규칙)로 강제한다. |
| AX-07 | **게이트는 망각에 대한 백스톱이지, 적대적 공격에 대한 샌드박스가 아니다.** 게이트의 위협 모델은 "지침을 잊거나 잘못 판단한 정직한 에이전트"다. 악의적 우회를 막는 것은 OS·컨테이너·권한 수준 격리의 일이며, 게이트로 그것을 흉내 내려 하면 복잡도만 폭발한다. |
| AX-08 | **모든 루프에는 정지 조건이 있다.** 반복 횟수 상한, 진전 없음 감지, wall-clock 상한 중 최소 하나가 없는 반복 구조는 기능이 아니라 버그다. |
| AX-09 | **실패는 시끄럽게, 성공은 조용하게.** 게이트·스크립트·검증기의 내부 오류가 조용히 "통과"로 해석되는 경로가 있어서는 안 된다. 판정 불가 상황은 명시적으로 fail-open인지 fail-closed인지 선언한다. |
| AX-10 | **사람은 가장 비싼 자원이므로, 인간 게이트는 비가역 지점에만 둔다.** 모든 단계에 사람 확인을 요구하면 사람이 러버스탬프가 된다. 사람의 판단은 스펙 승인과 비가역 실행 승인에 집중시킨다. |
| AX-11 | **검증 기준은 런 중에 동결된다.** 기준을 통과하려는 손과 기준을 고치는 손이 같아서는 안 된다. 런 도중 수용 기준·테스트를 약화시키는 변경은 기계적으로 차단하거나 최소한 기록·표면화한다. |
| AX-12 | **하네스 자체도 소프트웨어다.** 테스트되지 않은 게이트는 없는 게이트다. 게이트·파서·검증기는 골든 테스트와 회귀 스위트를 가진다. |

## Part 1 — 멘탈 모델과 설계 원리

### 1.1 정의 (PR-01)

- **하네스(harness)**: 확률적 코어(LLM) 주위를 감싸는 **결정론적 껍데기**.
  구성 요소는 다섯 가지다: 도구(tools), 정책(policy: 게이트·훅·권한),
  상태(state: 디스크 파일), 검증(verification: 독립 checker), 복구(recovery: 되돌림·재시작 수단).
- **에이전트(agent)**: 모델 + 도구 실행 능력을 가진 하나의 실행 단위. 하나의 컨텍스트 윈도우를 가진다.
- **루프(loop)**: 하네스 위에서 "만들기 → 채점 → 수정"을 반복하는 제어 구조. 루프는 하네스의
  소비자이지 하네스의 일부가 아니다 — 하네스가 견고하면 루프는 단순해진다.

핵심 관점: **하네스 엔지니어링은 프롬프트 엔지니어링이 아니라 시스템 엔지니어링이다.**
모델을 더 똑똑하게 만드는 것이 아니라, 모델이 멍청한 순간에도 시스템이 안전하고
관측 가능하게 실패하도록 만드는 것이다.

### 1.2 신뢰 경계 모델 (PR-02)

무엇을 신뢰하고 무엇을 검증하는가를 명시적으로 그린다. 기본 지도는 다음과 같다.

| 대상 | 신뢰 수준 | 근거·처리 |
|------|-----------|-----------|
| 하네스 코드 (게이트 스크립트, 파서) | 신뢰 | 사람이 리뷰하고 테스트한 코드. 단, AX-12에 따라 자체 테스트 필수. |
| 검증 명령의 exit code·출력 | 신뢰 | 기계 관측. 단, 검증 명령 자체가 런 중 변경되지 않았을 때만 (AX-11). |
| 디스크의 상태 파일 **구조** | 신뢰 | 파싱은 한다. 깨진 구조는 명시적 오류로 처리. |
| 디스크의 상태 파일 **내용** (모델이 쓴 것) | 조건부 | "무엇을 했다"는 기록은 참고용. 완료 여부 판정에는 쓰지 않고, 검증 명령으로 재확인. |
| 모델의 자기 보고 ("완료했습니다") | 불신 | AX-01. 판정 근거로 사용 금지. |
| maker가 첨부한 설명·증거 요약 | 불신 | checker는 maker의 요약이 아니라 원본(diff, 로그, 직접 실행 결과)을 본다. |
| 외부 콘텐츠 (웹, 서드파티 출력) | 불신 | 지침으로 해석 금지, 데이터로만 취급. |

설계 점검 시 이 표를 대상 시스템에 그려보고, "불신" 칸의 대상이 판정 경로에
끼어드는 지점이 있으면 그것이 결함이다.

### 1.3 자율성 결정 함수 (PR-03)

AX-05를 운영 규칙으로 내리면 2×2 매트릭스가 된다.

| | **되돌리기 쉬움** (git revert, 파일 복원) | **되돌리기 어려움** (배포, 외부 발신, 삭제) |
|---|---|---|
| **검증기 신뢰도 높음** (자동 테스트가 실패를 잡음) | **완전 자율.** 사람 개입 없이 반복. 루프의 홈그라운드. | **기계 게이트 + 사람 승인.** 검증이 통과해도 실행 직전 확인. |
| **검증기 신뢰도 낮음** (테스트 없음, 주관적 품질) | **자율 + 사후 리뷰.** 진행하되 diff·journal을 사람이 나중에 훑을 수 있게. | **금지 또는 전면 수동.** 이 사분면에서 에이전트를 돌리는 것 자체가 설계 오류. |

실무 규칙:
- 자율성을 한 칸 올리고 싶으면 프롬프트를 고치지 말고 **왼쪽·위로 이동시킬 투자**를 한다:
  테스트를 추가해 검증기 신뢰도를 올리거나, worktree·백업·dry-run으로 되돌림 비용을 낮춘다.
- 작업 하나가 여러 사분면에 걸치면(예: 코드 수정 + 배포), 사분면 경계에서 작업을 쪼개고
  경계에 게이트를 둔다.

### 1.4 fail-open vs fail-closed 판단표 (PR-04)

게이트 자체가 오류를 내거나 판정 불가일 때 어느 쪽으로 넘어질지는 게이트마다
**명시적으로 선언**되어야 한다 (AX-09). 판단 기준:

| 조건 | 방향 | 이유 |
|------|------|------|
| 게이트가 보호하는 대상이 비가역 (삭제, 배포, 발신, 과금) | **fail-closed** (판정 불가 → 차단) | 미탐 비용이 오탐 비용을 압도. 한 번의 통과 실수가 복구 불능. |
| 게이트가 흐름 제어용이고 대상이 가역 (재프롬프트, 상태 신선도 확인) | **fail-open** (판정 불가 → 통과) | 오탐이 누적되면 사람이 게이트를 끄게 되고, 꺼진 게이트는 없는 게이트다. |
| 승인 마커·자격 증명 검사 | **fail-closed** | 마커 파일이 깨졌거나 읽기 실패 → 승인 없음으로 간주. |
| 새로 도입한 게이트의 시험 운영 기간 | **fail-open + 로그** | 오탐률을 측정한 뒤 fail-closed로 승격. |

점검 시: 각 게이트 스크립트에서 오류 경로(파일 없음, 파싱 실패, 타임아웃)를 추적해
실제로 어느 쪽으로 넘어지는지 확인한다. **선언과 구현이 다르면 결함**이고,
선언 자체가 없으면 그것도 결함이다.

### 1.5 인간 게이트의 최소 배치 (PR-05)

사람의 판단이 실제로 필요한 지점은 세 곳뿐이다:

1. **입구 — 스펙 승인**: "무엇이 done인가"의 정의. 수용 기준을 기계 검증 가능한 형태로
   컴파일한 결과를 사람이 승인한다. 여기서 사람이 하는 일은 코드 리뷰가 아니라 **의도 확인**이다.
2. **중간 — 비가역 실행 승인**: G3 등급 행위(§3.6) 직전. "이 명령을 지금 실행해도 되는가."
3. **출구 — 머지/릴리스**: 최종 산출물의 채택 여부.

이 세 곳 외의 모든 확인 요청은 자동화 실패의 증상이다. "이 파일을 수정할까요?",
"테스트를 돌릴까요?" 같은 질문이 반복되면 하네스가 아니라 비서를 만든 것이다 (AP-05).

### 1.6 정보 비대칭 원리 (PR-06)

checker가 maker보다 아는 것이 하나도 없으면, checker는 maker의 서사를 재확인하는
것밖에 할 수 없다. 채점의 강도는 checker가 독점하는 정보의 양에 비례한다.

정보 비대칭을 만드는 수단 (약한 것 → 강한 것):

1. **독립 컨텍스트**: checker가 maker의 대화 이력을 보지 못한다. 최소 요건.
2. **원본 스펙 직접 접근**: checker는 maker의 해석이 아니라 승인된 spec 파일을 직접 읽는다.
3. **직접 실행**: checker가 검증 명령을 자기 손으로 실행한다 (maker의 실행 로그를 믿지 않는다).
4. **holdout**: maker에게 공개되지 않은 검증 세트(숨겨진 테스트, 별도 시나리오)를 checker만 보유.
   maker가 기준에 과적합(overfit)하는 것을 잡는 유일한 구조적 수단.

설계 점검 시 대상 시스템이 1–4 중 어디까지 확보했는지 판정한다. 1조차 없으면 AP-02다.

## Part 2 — 아키텍처

### 2.1 레이어드 아키텍처 (AR-01)

제로부터 만든다면 다음 여섯 계층으로 쌓는다. 화살표는 의존 방향이다
(위 계층은 아래를 관측·통제하고, 아래는 위를 알지 못한다).

```
┌──────────────────────────────────────────────────────┐
│ L6. 인간 게이트     스펙 승인 · G3 실행 승인 · 머지     │
├──────────────────────────────────────────────────────┤
│ L5. 검증 계층       독립 checker · holdout · verdict   │
├──────────────────────────────────────────────────────┤
│ L4. 정책 계층       게이트 · 훅 · 권한 규칙 · 동결     │
├──────────────────────────────────────────────────────┤
│ L3. 상태 계층       디스크 상태 · journal · 카운터     │
├──────────────────────────────────────────────────────┤
│ L2. 도구 계층       파일 I/O · 셸 · 검색 · 격리 실행   │
├──────────────────────────────────────────────────────┤
│ L1. 모델 코어       LLM (확률적 · 신뢰 불가)           │
└──────────────────────────────────────────────────────┘
```

계층별 불변식:

| 계층 | 불변식 | 위반 시 증상 |
|------|--------|--------------|
| L1 모델 | 모델은 자신의 완료 여부를 선언할 수 없다 — 선언은 L5의 판정을 인용할 때만 유효. | 낙관 보고 (AP-09) |
| L2 도구 | 모든 부수효과는 도구 호출을 경유한다 — 도구를 우회하는 부수효과 경로 없음. | 게이트가 볼 수 없는 변경 |
| L3 상태 | 새 프로세스가 L3만 읽고 재개 가능. 컨텍스트에만 있는 결정 사항 없음. | 세션 재시작 시 작업 증발 (AP-03) |
| L4 정책 | 게이트는 도구 호출 경로 위에 있어 우회 불가(정직한 에이전트 기준). 모델은 L4 코드를 수정할 수 없거나, 수정 자체가 게이트 대상. | 게이트가 장식이 됨 |
| L5 검증 | checker는 읽기 전용 + 독립 컨텍스트. maker 산출물에 의존하지 않는 자체 실행 경로 보유. | 러버스탬프 (AP-01) |
| L6 인간 | 사람에게 도달하는 요청은 옵션과 근거가 정리된 결정 요청뿐. 원시 로그 던지기 금지. | 사람이 병목·러버스탬프화 |

### 2.2 제어 평면 / 데이터 평면 분리 (AR-02)

- **데이터 평면**: 모델이 만드는 산출물 — 소스 코드, 문서, 설정. 모델이 자유롭게 쓴다.
- **제어 평면**: 하네스가 판단에 쓰는 것 — spec, verdict, 승인 마커, 게이트 스크립트, 카운터.

쓰기 권한 매트릭스:

| 파일 | 모델(maker) | checker | 하네스 스크립트 | 사람 |
|------|:---:|:---:|:---:|:---:|
| 데이터 평면 (소스 등) | ✅ | ❌ (읽기만) | — | ✅ |
| state (진행 상태) | ✅ 갱신 | ❌ | 검사 | ✅ |
| journal (이력) | ✅ append-only | ❌ | append | ✅ |
| spec (수용 기준) | ⛔ 런 중 동결 (강화만 예외 허용 가능) | ❌ | 검사 | ✅ 승인 주체 |
| verdict (판정) | ⛔ | ✅ 산출 | 검사 | — |
| 승인 마커 | ⛔ | ❌ | 소비(1회용) | ✅ 발행 |
| 게이트 스크립트 | ⛔ 또는 게이트 대상 | ❌ | — | ✅ |

점검 시 이 매트릭스의 ⛔ 칸이 실제로 강제되는지(권한 규칙·훅으로), 아니면
지침으로만 존재하는지(AX-06 위반) 확인한다. 모델이 verdict를 쓸 수 있는 시스템은
채점이 존재하지 않는 시스템이다.

### 2.3 상태 아키텍처 (AR-03)

- **단일 진실 디렉토리**: 가변 상태 전체를 한 디렉토리 아래에 모은다 (§3.1).
  상태가 여러 곳에 흩어지면 재개 절차가 "디렉토리 하나 읽기"에서 "고고학"이 된다.
- **재시작 복구 테스트**: 아키텍처의 수용 기준 그 자체 — *"세션을 지금 kill하고
  새 세션을 열었을 때, 새 세션이 상태 디렉토리만 읽고 다음 액션을 정확히 결정할 수 있는가."*
  이 테스트를 통과 못 하면 상태 설계가 미완이다.
- **컨텍스트 주입**: 세션 시작 시점에 상태 요약(digest)을 자동 주입하는 훅을 둔다.
  규칙: **디스크 상태가 컨텍스트 기억보다 우선한다**를 digest 머리에 명시한다 —
  요약된 과거 기억과 디스크가 충돌할 때 모델이 디스크를 따르게 하기 위함이다.
- **쓰기 규율**: 상태 갱신은 "결정 직후"에 한다. 턴 끝에 몰아 쓰면 크래시 시 결정이 증발한다.

### 2.4 격리 아키텍처 (AR-04)

격리 수단은 위협에 맞춰 고른다. 과잉 격리는 마찰이고, 과소 격리는 사고다.

| 위협 | 격리 수단 | 선택 기준 |
|------|-----------|-----------|
| 병렬 작업 간 파일 충돌 | **git worktree** (작업별 격리 트리) | 둘 이상의 에이전트가 같은 레포를 동시에 수정할 때. 단일 에이전트 단일 작업이면 불필요. |
| 위험 명령의 시스템 피해 | **샌드박스 / 컨테이너 / 권한 축소** | 신뢰 불가 코드 실행, 시스템 전역을 건드릴 수 있는 명령. AX-07 — 이것이 적대적 위협의 담당자다. |
| 외부 부수효과 (API 호출, 발신, 과금) | **allowlist + dry-run + G2/G3 게이트** | 네트워크 밖으로 나가는 모든 것. dry-run 모드가 없는 외부 연동은 만들지 않는다. |
| 실험적 변경의 본선 오염 | **브랜치 규율** (main 직접 작업 금지 게이트) | 항상. 비용이 거의 0인 격리는 기본값으로 켠다. |

### 2.5 관측성 최소 세트 (AR-05)

"무슨 일이 있었는지 파일만 보고 재구성 가능"이 기준이다. 최소 네 가지:

1. **journal** (append-only): 결정과 그 이유. "무엇을 했다"보다 "왜 그쪽을 골랐다"가 값지다 —
   다음 세션이 같은 갈림길에서 같은 실수를 반복하지 않게 한다.
2. **verdict 기록**: 매 채점의 per-criterion 결과 + 증거. 덮어쓰지 않고 런별로 보존.
3. **카운터**: iteration 수, wall-clock, 토큰/비용 추정. 정지 조건(AX-08)의 입력값이므로
   관측이 아니라 제어의 전제 조건이다.
4. **게이트 발화 로그**: 어떤 게이트가 언제 무엇을 차단/통과시켰는가. 오탐률 측정과
   fail-open→fail-closed 승격 판단(PR-04)의 근거 데이터.

### 2.6 컨텍스트 아키텍처 (AR-06)

컨텍스트 윈도우는 캐시(AX-04)이자 **희소 자원**이다. 하네스는 컨텍스트를 예산처럼 배분한다.

- **메인 컨텍스트의 용도를 제한한다**: 오케스트레이션, 결정, 사용자 소통.
  대량 파일 읽기·광역 탐색·장문 로그 분석은 서브에이전트로 격리한다 — 메인은
  결론만 받는다. 메인 컨텍스트에 원문을 쌓는 것은 캐시에 콜드 데이터를 채우는 것이다.
- **서브에이전트 반환 규약**: 결론 + 증거 포인터(파일:라인, 명령, exit code)만.
  원문 덤프 반환 금지. 이 규약은 서브에이전트 프롬프트에 명시한다.
- **요약(compaction) 생존 설계**: 긴 세션은 언젠가 요약된다. 요약 후에도 살아남아야
  하는 사실(결정, 미해결 항목, 다음 액션)은 요약을 신뢰하지 말고 디스크에 있어야
  한다 — AR-03의 쓰기 규율이 이것의 보험이다.
- **세션 시작 digest**: 시작 훅이 상태 파일의 요약을 주입한다. digest 머리에
  "디스크 상태가 컨텍스트 기억보다 우선"을 명시 (AR-03).
- **오염 관리**: 실패한 시도의 장황한 출력은 이후 판단을 오염시킨다.
  실패는 journal에 사유와 함께 한 줄로 남기고, 컨텍스트에서는 요약으로 대체한다.

### 2.7 멀티에이전트 토폴로지 (AR-07)

에이전트를 병렬로 늘리는 것은 공짜가 아니다 — 조정 비용과 충돌 위험이 함께 는다.

- **기본 토폴로지는 orchestrator–worker**: worker끼리는 통신하지 않고,
  orchestrator만 결과를 종합한다. peer-to-peer 통신은 디버깅 불가능한
  상태 공간을 만든다 — 필요가 증명되기 전에는 도입하지 않는다.
- **병렬화 판정 기준** (둘 다 충족할 때만 fan-out):
  1. 작업 간 쓰기 충돌이 없다 — 있으면 worktree 격리(AR-04)가 선행 조건.
  2. 결과 종합이 기계적으로 가능하다 (합집합, 최댓값, 독립 판정의 다수결).
     "종합하려면 사람이 다시 다 읽어야 함"이면 병렬화 이득이 없다.
- **worker 프롬프트는 자기완결**: worker는 orchestrator의 대화 이력을 공유하지
  않는다고 가정하고, 필요한 컨텍스트를 전부 프롬프트에 담는다.
- **공유 상태의 단일 작성자**: `.harness/` 상태 파일에는 orchestrator만 쓴다.
  worker가 공유 상태에 직접 쓰면 마지막 쓰기가 이긴다(lost update).
- **채점 fan-out** (PR-06 강화의 병렬 형태): 판정이 갈릴 수 있는 검증은
  독립 checker N개의 다수결로. 단, 같은 프롬프트의 복제 N개보다
  **관점을 달리한 N개**(정확성/보안/재현성)가 서로 다른 실패 모드를 잡는다.

## Part 3 — 구현

이 파트가 "제로부터 만든다면 정확히 어떻게"에 대한 답이다. 예시는 셸 + 마크다운 기반
(가장 이식성 높은 조합)이지만, 원칙은 언어 무관이다.

### 3.1 표준 디렉토리 구조 (IM-01)

```
<project>/
├── .harness/                  # 가변 상태의 단일 진실 (AR-03)
│   ├── config.md              # 사람이 쓰는 설정. 런 중 동결.
│   ├── spec.md                # 승인된 수용 기준. 런 중 동결 (AX-11).
│   ├── state.md               # 현재 상태. maker가 갱신, 게이트가 검사.
│   ├── journal.md             # append-only 이력 (AR-05).
│   └── runs/<run-id>/         # 런별 산출물
│       ├── verdict.md         # checker의 판정 (checker만 산출)
│       └── gate.log           # 게이트 발화 로그
├── harness/                   # 하네스 코드 (불변 로직 — 모델 수정 금지 대상)
│   ├── gates/
│   │   ├── lib.sh             # 공용 파싱 — 단일 소스 (IM-03)
│   │   ├── action_gate.sh     # 도구 호출 전 게이트
│   │   ├── stop_gate.sh       # 정지/완료 선언 게이트
│   │   └── freeze_gate.sh     # 기준 동결 게이트
│   └── tests/
│       ├── run.sh             # 골든 테스트 러너 (IM-07)
│       └── fixtures/          # 가짜 도구 호출 입력 JSON
└── (데이터 평면: 소스 코드 등)
```

구조적 결정의 이유:
- **상태와 로직의 물리적 분리**: `.harness/`(가변)와 `harness/`(불변)를 나누면
  "모델이 써도 되는 곳"의 경계가 디렉토리 경계와 일치해 게이트 조건이 한 줄이 된다.
- **runs/ 하위 보존**: verdict를 단일 파일로 덮어쓰면 "이전 판정 대비 진전"을 잃는다.
  런별 디렉토리는 no-progress 감지(IM-05)의 데이터 소스다.

### 3.2 상태 파일 포맷 명세 (IM-02)

**포맷 선택 원칙**: 모델이 깨뜨리기 어렵고, 셸에서 grep 한 줄로 파싱되고,
git diff로 사람이 리뷰 가능한 형식. → **한 줄에 하나의 사실**을 담는
`key: value` 라인과 체크박스 라인. YAML/JSON 중첩 구조는 피한다 —
모델의 부분 편집(Edit)에 깨지기 쉽고, 셸 게이트에서 파서 의존성이 생긴다.

**state.md**:

```markdown
# State
run_active: true
phase: implement          # implement | verify | escalated | done
iteration: 3
last_verdict: 2/5         # checker만이 근거를 만들 수 있는 값

## Next action
- R3 실패 원인: 파서가 빈 입력에서 crash. tests/parser_test.sh:12 재현.

## Decisions
- 스트리밍 파서 대신 2-pass 채택 — 입력이 항상 <1MB (journal 2026-07-11 참조)
```

**spec.md** — 수용 기준은 **반드시** 검증 명령을 동반한다 (AX-02):

```markdown
# Spec
approved: kim 2026-07-11   # 승인 전 approved: pending

## Criteria
- [ ] R1: 빈 입력에서 exit 0 — verify: echo "" | ./parse && echo OK
- [ ] R2: 골든 출력 일치 — verify: diff <(./parse fixtures/a.in) fixtures/a.out
- [ ] R3: 전체 테스트 green — verify: bash tests/run.sh

## Escalated — not machine-verifiable
- "에러 메시지가 친절할 것" → 판정 불가. 옵션: (a) 골든 메시지 고정 (b) 기준에서 제외. 사람 결정 대기.
```

핵심 규칙:
- **verify: 없는 기준은 기준이 아니다.** 기계 검증 불가 요구는 조용히 해석하지 말고
  `Escalated` 섹션에 옵션과 함께 격리한다 (PR-05 입구 게이트의 실체).
- **verdict.md** (checker 산출) — 증거 필수:

```markdown
# Verdict
run_id: 2026-07-11-a
result: 2/3
- R1: PASS — 직접 실행, exit 0, 출력 "OK"
- R2: PASS — diff exit 0
- R3: FAIL — tests/run.sh exit 1: parser_test.sh line 12 "empty input crash"
```

- verdict의 각 행은 **checker가 직접 실행한 명령의 관측 결과**여야 한다.
  "maker의 로그에 따르면"으로 시작하는 증거는 증거가 아니다 (PR-06).

### 3.3 게이트·훅 구현 패턴 (IM-03)

게이트는 도구 호출 경로에 끼어드는 짧은 스크립트다. 설계 규칙:

- **exit code 규약을 호스트 런타임에 맞추되 한 곳에 문서화한다.**
  일반형: `0` = 허용, 지정된 비0 코드 = 차단 + 사유를 지정 스트림으로.
  차단 사유 메시지는 모델이 읽고 스스로 교정할 수 있도록 "무엇이, 왜, 대안은"을 담는다.
- **파싱은 lib 하나로.** 도구 호출 JSON에서 명령·경로를 꺼내는 코드가 게이트마다
  복사되면 게이트마다 다르게 뚫린다. 공용 `lib.sh` 단일 소스 (AX-12의 테스트 대상도 이것 하나).
- **짧고 보수적으로.** 게이트당 실행 시간 < 1초 목표, 외부 네트워크 호출 금지,
  판정 불가 시의 방향(fail-open/closed)을 파일 머리 주석에 선언 (PR-04).
- **멱등**: 같은 입력에 같은 판정. 게이트 안에서 상태를 변경하지 않는다
  (예외: 발화 로그 append, 1회용 마커 소비).

**패턴 A — action gate** (도구 호출 전, 비가역 명령 차단):

```sh
#!/bin/sh
# fail-closed: 파싱 불가 명령은 차단 (보호 대상이 비가역이므로)
cmd=$(parse_tool_command "$INPUT_JSON") || deny "명령 파싱 실패"

if is_g2_or_above "$cmd"; then          # §3.6의 등급 판정
  if consume_approval_marker; then       # 1회용 마커 (아래 패턴 C)
    allow
  else
    deny "G2+ 명령: $cmd — 사람 승인 마커 필요. 발행: harness/approve.sh"
  fi
fi
allow
```

**패턴 B — stop gate** (완료 선언 검사; AX-02의 강제 지점):

```sh
#!/bin/sh
# fail-open: 흐름 제어용 게이트. 상태 파일이 없거나 필드가 없으면 통과.
phase=$(field state.md phase) || allow
[ "$phase" = "done" ] || allow

verdict=".harness/runs/$(field state.md run_id)/verdict.md"
[ -f "$verdict" ]                 || deny "done 선언인데 verdict 없음 — checker 먼저"
[ "$verdict" -ot ".harness/state.md" ] && deny "verdict가 마지막 변경보다 오래됨(stale) — 재채점 필요"
passed_all "$verdict"             || deny "verdict $(field "$verdict" result) — 미통과 기준 존재"
allow
```

stale 판정에 주목: **verdict는 그것이 채점한 산출물보다 새것일 때만 유효하다.**
파일 mtime 비교(`-ot`)가 가장 단순한 구현이다. 채점 후 코드를 또 고쳤다면 그 verdict는 무효다.

**패턴 C — 승인 마커** (인간 게이트의 기계 인터페이스):

- 사람이 별도 명령으로 마커 파일을 발행한다 (예: `.harness/.approved-g2`).
- 게이트는 마커를 **검증 후 즉시 삭제**한다 — 1회용. 남겨두면 승인 하나가
  무한 승인이 된다.
- 마커 읽기 실패·형식 불일치 → 승인 없음 (fail-closed, PR-04).
- 강화 옵션: 마커에 승인 대상 명령의 해시를 넣어 "그 명령에 대한 승인"으로 좁힌다.

**패턴 D — freeze gate** (AX-11의 강제 지점):

```sh
#!/bin/sh
# run_active일 때만 발동. 비런 세션 오탐 없음.
[ "$(field state.md run_active)" = "true" ] || allow
target=$(parse_edit_target "$INPUT_JSON")
case "$target" in
  */spec.md)
    edit_weakens_criteria "$INPUT_JSON" \
      && deny "런 중 spec 약화 금지 — 기준 추가(강화)만 허용" ;;
  */harness/gates/*)
    deny "런 중 게이트 코드 수정 금지" ;;
esac
allow
```

"약화"의 기계 판정: 기존 `verify:` 라인의 변경·삭제 = 약화(차단), 새 기준 라인 추가 = 강화(허용).
기계로 판정 불가한 동결(예: "테스트를 회피성으로 고쳤는가")은 게이트에 넣지 말고
checker의 채점 항목이나 사후 리뷰로 보낸다 (AP-10).

### 3.4 검증기(checker) 명세 (IM-04)

checker는 하네스에서 가장 투자 가치가 높은 컴포넌트다 (AX-05의 첫째 인수).

**필수 속성**:
1. **독립 컨텍스트**: maker의 대화 이력이 없는 새 에이전트/프로세스로 실행.
2. **읽기 전용**: 파일 수정 도구를 갖지 않는다. 도구 목록 자체에서 제거한다 —
   "수정하지 마라"는 지침이 아니라 능력의 부재로 강제 (AX-06).
3. **직접 실행**: spec의 각 `verify:` 명령을 자기 손으로 실행하고 exit code를 관측.
4. **원본 접근**: spec.md를 직접 읽는다. maker가 요약해 준 기준을 쓰지 않는다.

**판정 규칙**:
- 기준마다 PASS/FAIL + 증거 한 줄 (실행한 명령, exit code, 핵심 출력).
- **판정 불가 = FAIL.** verify 명령이 실행 안 되면, 모호하면, 타임아웃이면 FAIL로 기록하고
  사유를 남긴다. "아마 되는 것 같다"는 판정이 아니다.
- verdict 산출 외 어떤 부수효과도 없다.

**강화 단계** (여력이 생기는 순서대로):
- **cross-model**: maker와 checker를 다른 모델로. 같은 모델은 같은 맹점을 공유한다.
- **holdout** (PR-06): checker 전용 검증 세트. maker의 기준 과적합을 잡는 유일한 구조적 수단.
- **적대적 checker**: "통과시켜라"가 아니라 "반증하라"로 프롬프트한 별도 인스턴스를 병행.

### 3.5 정지 조건과 에스컬레이션 (IM-05)

AX-08의 구현. 루프에는 **네 겹**의 정지 조건을 겹쳐 둔다 — 하나로는 부족하다:

| 조건 | 구현 | 기본값 제안 |
|------|------|-------------|
| 반복 상한 | state.md의 `iteration` 카운터, 게이트가 검사 | 작업당 5–10회 |
| 진전 없음 | 직전 run과 이번 run의 verdict 비교 — 동일 기준이 동일 사유로 연속 FAIL | 2연속 → 전략 변경 강제, 3연속 → 정지 + 에스컬레이션 |
| wall-clock | run 시작 timestamp를 state에 기록, 게이트가 경과 검사 | 작업 성격에 따라 명시적 설정 (기본값 없이 통과 금지) |
| 예산 | 토큰/비용 카운터 상한 | 사람이 config에 명시 |

**에스컬레이션 프로토콜** — 정지 조건 발동 시 에이전트가 하는 일:
1. state.md의 phase를 `escalated`로 바꾸고 **멈춘다** (조용히 계속하지 않는다).
2. journal에 기록: 무엇을 시도했고, 무엇이 반복 실패했고, 왜 막혔는가.
3. 사람에게 전달할 **결정 요청**을 만든다: 상황 요약 + 옵션 2–3개 + 각 옵션의 트레이드오프.
   원시 로그를 던지는 것은 에스컬레이션이 아니라 책임 전가다 (L6 불변식).

### 3.6 비가역성 등급과 게이트 정책 (IM-06)

모든 부수효과를 네 등급으로 분류하고, 등급이 게이트 정책을 결정한다.
**판정 기준은 "되돌리는 데 무엇이 필요한가"** 하나다.

| 등급 | 정의 (되돌림 수단) | 예시 | 게이트 정책 |
|------|--------------------|------|-------------|
| **G0** | 도구 한 번 (파일 복원, git checkout) | 워킹트리 편집, 읽기, 테스트 실행 | 없음 — 자유 |
| **G1** | 명령 한 번 (revert, branch 삭제) | 로컬 커밋, 피처 브랜치 push | journal 기록 후 자동 진행 |
| **G2** | 복잡한 복구 또는 부분 손실 | force push, 히스토리 변경, 대량 삭제, 공유 상태 변경 | 기계 게이트 차단 + 1회용 승인 마커로만 통과 |
| **G3** | 복구 불능 또는 외부 관측됨 | main 머지, 배포, 이메일·API 발신, 과금, 데이터 파기 | 인간 게이트 — 사람 승인 없이는 경로 자체가 없음 |

운영 규칙:
- 등급 분류표는 하네스 코드(action gate) 안에 패턴으로 존재해야 한다.
  문서에만 있는 분류는 분류가 아니다 (AX-06).
- 애매하면 한 등급 올린다. 등급을 내리는 것은 오탐 데이터가 쌓인 뒤에 한다 (AR-05-4).
- G3의 "외부 관측됨"에 주의: 기술적으로 삭제 가능해도(보낸 메시지 삭제 등)
  이미 관측되었을 수 있으면 비가역으로 분류한다.

### 3.7 하네스 자체의 테스트 (IM-07)

AX-12의 구현. 세 겹:

1. **골든 테스트**: 각 게이트에 가짜 도구 호출 입력(fixture JSON)을 주입하고
   기대 exit code·메시지를 단언한다. 커버해야 할 케이스:
   - 정상 통과 / 정상 차단 (등급별 대표 명령)
   - **오류 경로**: 상태 파일 없음, 필드 없음, 파싱 실패, stale — 각각이 선언된
     fail 방향으로 넘어지는지 (PR-04의 "선언 = 구현" 검증)
   - 승인 마커: 유효 / 없음 / 사용 후 재사용 시도
2. **적대적 셀프 리뷰**: 주기적으로 red-team 에이전트에게 "이 게이트들을 정상 도구
   사용만으로 우회해 보라"를 시킨다. 발견된 구멍 중 **정직한 에이전트가 실수로 밟을 수
   있는 것**만 고친다 — 의도적 악용 시나리오는 AX-07에 따라 격리 계층의 문제로 넘긴다.
3. **단일 소스 회귀**: CI가 도는 체크와 로컬 검증 스크립트가 같은 것을 실행한다.
   "로컬은 green인데 CI는 red"는 검증 이중화의 증상이다.

### 3.8 도구·명령 설계 규칙 (IM-08)

에이전트가 쓰는 도구·스크립트·CLI는 사람용과 설계 기준이 다르다. 에이전트는
문서를 안 읽고 시행착오로 배우므로, **도구의 반응 자체가 문서**여야 한다.

1. **오류는 다음 행동을 알려준다.** 실패 메시지는 "무엇이 틀렸고, 올바른 형태는
   무엇인가"를 담는다. `Error: invalid input`은 재시도 루프를 만들고,
   `Error: --format은 json|md 중 하나. 받은 값: yaml`은 한 번에 교정시킨다.
2. **침묵 실패 금지.** 실패는 반드시 비0 exit code. "경고만 출력하고 exit 0"은
   에이전트에게는 성공이다 (AX-09의 도구 버전).
3. **멱등이거나, 비멱등을 표시하거나.** 에이전트는 타임아웃·오류 시 재시도한다.
   재실행이 안전하지 않은 명령은 이름·도움말에 명시하고 G1 이상으로 분류한다 (IM-06).
4. **외부 부수효과 도구에는 dry-run이 있다.** dry-run 없는 외부 연동은 만들지
   않는다 (AR-04). 에이전트의 기본 경로가 dry-run이고, 실제 실행이 명시적 플래그가
   되도록 방향을 잡는다.
5. **출력은 기계 소비 우선.** 판정에 쓰일 출력은 grep 가능한 안정된 형식으로.
   장식(색상, 진행 바, 박스 문자)은 파싱을 깨뜨린다 — TTY 감지로 끈다.
6. **도구 수는 최소로, 이름은 서로 멀게.** 비슷한 이름·기능의 도구가 나열되면
   오호출률이 오른다. 하나로 합치거나, 언제 무엇을 쓰는지 도구 설명에 판정
   기준을 넣는다.

### 3.9 지침 파일 설계 (IM-09)

지침 파일(시스템 프롬프트에 주입되는 프로젝트 문서)은 하네스의 일부이지만,
가장 약한 계층(L1에 대한 요청)임을 잊지 않는다.

- **지도이지 백과사전이 아니다.** 지침 파일에는 "무엇이 어디에 있고, 언제 무엇을
  읽어라"만. 상세 절차·장문 규칙은 별도 문서로 옮기고 링크한다. 모든 세션에
  주입되는 파일의 한 줄은 그만큼의 상시 컨텍스트 비용이다.
- **prompt vs hook 결정표** (AX-06의 적용 기준):

  | 규칙의 성격 | 배치 |
  |------------|------|
  | 위반이 비가역 피해 (G2/G3 행위) | 훅·게이트 — 지침에는 "게이트가 있다"는 사실만 |
  | 위반이 가역이지만 매번 지켜야 함 (포맷, 검증 절차) | 훅으로 검사 가능하면 훅, 아니면 지침 + checker 채점 항목 |
  | 스타일·선호·판단 가이드 | 지침 |
- **이유를 함께 적는다.** "X를 하지 마라"보다 "X를 하지 마라 — Y가 깨진다"가
  변형 상황에서 올바르게 일반화된다. 이유 없는 규칙은 예외 상황에서 무시되거나
  과잉 적용된다.
- **금지 목록보다 올바른 경로.** 부정 지시 나열은 컨텍스트에서 희석된다.
  올바른 기본 경로를 하나 제시하는 쪽이 준수율이 높다.
- **지침 파일 자체를 기계 검사한다**: 크기 예산(줄 수 상한), 링크 유효성,
  "절대/반드시" 규칙이 대응 훅을 갖는지(CK-D-05)를 CI에 넣는다.

### 3.10 제로부터 구축 순서 (IM-10)

하네스를 제로부터 만들 때의 순서. 각 단계는 그 자체로 배포 가능한 개선이며,
**완료 기준을 통과하기 전에는 다음 단계로 가지 않는다.** 순서의 논리:
상태 없이는 정지 조건을 만들 수 없고, spec 없이는 checker가 채점할 것이 없고,
checker 없이는 stop gate가 검사할 verdict가 없다 — 의존 순서가 곧 구축 순서다.

| 단계 | 구축물 | 완료 기준 (기계 검증) | 도달 |
|------|--------|----------------------|------|
| 1 | 상태 계층: `.harness/` 구조, state.md·journal.md 포맷 (IM-01, IM-02) | 재시작 복구 테스트 통과 — 세션 kill 후 새 세션이 상태 파일만 읽고 다음 액션 결정 (AR-03) | L2 진입 |
| 2 | 정지 조건: iteration 카운터 + 상한 검사 (IM-05) | 상한 초과 fixture에서 루프가 escalated로 전이하는 골든 테스트 green | L2 |
| 3 | spec 규율: verify: 필수 포맷 + Escalated 섹션 + 승인 필드 (IM-02) | verify: 없는 기준이 lint에 걸림 | 입구 확보 |
| 4 | checker: 독립 스폰 + 읽기 전용 + 직접 실행 + verdict 포맷 (IM-04) | 일부러 심은 실패 기준을 checker가 FAIL + 증거로 잡음 | L3 진입 |
| 5 | stop gate: done 선언 ↔ verdict 존재·신선도·전건 통과 연결 (IM-03 패턴 B) | verdict 없음/stale/부분 통과 fixture 각각 차단되는 골든 테스트 | L3 |
| 6 | 비가역 등급표 + action gate + 승인 마커 (IM-06, IM-03 패턴 A·C) | G2 대표 명령이 마커 없이 차단, 마커로 1회 통과, 재사용 차단 | L3 완성 |
| 7 | freeze gate: 런 중 spec·게이트 코드 약화 차단 (IM-03 패턴 D) | verify: 라인 삭제 fixture 차단, 기준 추가 fixture 통과 | L4 진입 |
| 8 | 관측성 완비: 게이트 발화 로그 + 카운터 → 오탐 데이터로 fail 방향 승격 (AR-05, PR-04) | 신규 게이트가 fail-open 시험 → 데이터 근거로 fail-closed 승격된 이력 1건 | L4 |
| 9 | (필요 시) holdout + 적대적 셀프 리뷰의 회귀 편입 (PR-06-4, IM-07-2) | red-team이 찾은 구멍이 fixture로 회귀 스위트에 존재 | L5 |

운영 원칙:
- **모든 신규 게이트는 fail-open + 로그로 태어난다** (PR-04). 오탐률을 보고 승격한다.
  첫날부터 fail-closed로 태어난 게이트는 첫 오탐에 꺼진다.
- 단계를 건너뛰고 싶은 유혹의 대부분은 "checker 없이 게이트부터"인데, 이는
  검사할 verdict가 없는 stop gate — 즉 장식을 만드는 것이다.
- 각 단계의 골든 테스트가 곧 하네스 회귀 스위트로 누적된다 (IM-07).

## Part 4 — 안티패턴 카탈로그

각 항목: **증상 → 왜 생기는가 → 교정**. 점검 시 발견하면 해당 AP ID로 인용한다.

**AP-01 · 러버스탬프 검증**
- 증상: checker가 maker의 리포트·요약만 읽고 PASS를 낸다. verdict에 "직접 실행" 증거가 없다.
- 원인: checker에게 실행 도구를 안 주었거나, 프롬프트가 "확인해 줘"라서 동의를 유도.
- 교정: IM-04 필수 속성 3(직접 실행)과 판정 규칙(판정 불가 = FAIL) 적용. verdict 포맷에 증거 칸을 강제.

**AP-02 · 자기 채점**
- 증상: 산출물을 만든 세션이 그대로 "검증도 완료했습니다"라며 done을 선언.
- 원인: 별도 checker 호출이 귀찮거나, 하네스에 stop gate가 없어 선언이 그냥 통과됨.
- 교정: AX-03 + IM-03 패턴 B — done 선언 경로에 독립 verdict 존재·신선도 검사를 기계로 끼움.

**AP-03 · 컨텍스트-온리 상태**
- 증상: 세션이 죽거나 압축되면 "지금까지 뭘 하기로 했는지"가 사라진다. 재시작 후 같은 탐색을 반복.
- 원인: 상태 기록이 "나중에 정리해야지"로 미뤄짐. 디스크 상태의 수용 기준(재시작 테스트) 부재.
- 교정: AR-03 — 결정 직후 쓰기 규율 + 재시작 복구 테스트를 하네스의 수용 기준으로 채택.

**AP-04 · 상한 없는 루프**
- 증상: 같은 테스트가 계속 실패하는데 밤새 같은 수정을 반복. 토큰만 소각.
- 원인: "언젠가 통과하겠지"라는 낙관이 구조에 반영됨. 진전 없음과 반복을 구분하는 장치 부재.
- 교정: IM-05의 네 겹 정지 조건. 특히 no-progress 감지(동일 기준 동일 사유 연속 FAIL)가 핵심.

**AP-05 · 사람 게이트 남발**
- 증상: "진행할까요?"가 단계마다 나온다. 사람은 내용을 안 읽고 y만 누르게 된다.
- 원인: 자율성의 근거(검증기·되돌림)에 투자하는 대신 책임을 사람에게 전가.
- 교정: PR-05 — 인간 게이트를 입구·G3·출구 세 지점으로 축소하고, 나머지는 PR-03 매트릭스의 왼쪽 위로 이동시키는 투자로 해결.

**AP-06 · 게이트를 샌드박스로 착각**
- 증상: "모델이 base64로 인코딩해서 우회하면?" 류의 시나리오에 대응해 게이트가 정규식 수십 개로 비대해짐. 오탐이 늘고 유지보수 불능.
- 원인: 위협 모델 미정의. 정직한 실수 방지(게이트의 일)와 악의 차단(격리의 일)의 혼동.
- 교정: AX-07 — 위협 모델을 문서에 명시하고, 적대적 시나리오는 샌드박스·권한 계층으로 이관. 게이트는 단순하게 유지.

**AP-07 · 프롬프트로 강제**
- 증상: CLAUDE.md류 지침 파일에 "절대 main에 push하지 마시오"가 있는데 가끔 push된다.
- 원인: 확률적 컴포넌트에 결정론적 책임을 부여함.
- 교정: AX-06 — "절대"가 붙는 모든 규칙을 훅·권한 규칙·게이트로 옮긴다. 지침 파일에는 "왜"와 지도만 남긴다.

**AP-08 · 런 중 기준 완화**
- 증상: 통과 못 하는 테스트가 skip 처리되거나, spec의 기준이 슬그머니 느슨해진 채 done 선언.
- 원인: maker에게 기준 수정 권한이 열려 있고 동결 게이트가 없음. 목표 달성 압력이 기준을 향함.
- 교정: AX-11 + IM-03 패턴 D. 기계 판정 가능한 약화(verify: 라인 변경·삭제)는 차단, 나머지는 checker 채점 항목으로.

**AP-09 · 검증 없는 낙관 보고**
- 증상: "구현 완료했고 테스트도 통과합니다"라는 보고 후 확인해 보면 테스트가 실행된 적 없음.
- 원인: 모델의 자기 보고가 판정 경로에 들어와 있음 (PR-02 신뢰 경계 위반).
- 교정: 보고 포맷에 관측 증거(명령 + exit code)를 강제하고, stop gate가 verdict 없는 done을 차단.

**AP-10 · 만능 게이트**
- 증상: "코드 품질이 낮으면 차단" 같은 판정 불가 조건이 게이트에 들어가 오탐 연발. 사람이 게이트를 끈다.
- 원인: 기계 판정 가능/불가의 경계를 긋지 않음.
- 교정: 게이트에는 기계 판정 가능한 조건만 (파일 경로, 명령 패턴, exit code, mtime). 주관적 품질은 checker·리뷰의 영역. 꺼진 게이트는 없는 게이트임을 상기 (PR-04).

**AP-11 · 침묵 fail**
- 증상: 게이트 스크립트가 내부 오류(파일 없음 등)로 죽었는데 런타임이 그것을 "통과"로 해석. 보호가 사라진 줄 아무도 모름.
- 원인: 오류 경로 미설계. fail 방향 미선언 (PR-04).
- 교정: AX-09 — 모든 오류 경로에 명시적 exit. 골든 테스트가 오류 경로를 커버 (IM-07-1).

**AP-12 · 이력 없는 판단**
- 증상: 다음 세션이 이전 세션과 같은 대안을 검토하고 같은 막다른 길에 들어간다.
- 원인: journal이 없거나 "무엇을 했다"만 기록하고 "왜 그쪽을 골랐다 / 무엇이 안 됐다"를 안 남김.
- 교정: AR-05-1 — 결정과 기각 사유를 journal에 기록하는 규율. state.md의 Decisions 섹션 (IM-02).

### 5.3 성숙도 등급 모델

하위 등급의 요건을 **전부** 충족해야 상위 등급을 부여한다. 하나라도 미충족이면
그 등급이 상한이다. 판정 시 "Lx — 미충족 요건: ..." 형식으로 보고한다.

| 등급 | 이름 | 요건 (누적) |
|------|------|-------------|
| **L1** | 프롬프트 의존 | 지침 파일은 있으나 기계 강제·디스크 상태 없음. (사실상 하네스 부재 — 출발점) |
| **L2** | 상태 확보 | 디스크 단일 진실 (CK-D-03 PASS) + 모든 루프에 정지 조건 (CK-D-04 PASS). |
| **L3** | 검증 분리 | maker/checker 분리 (CK-D-02) + 기계 검증 done (CK-D-01) + G2/G3 게이트 존재 (CK-D-06) + checker 직접 실행 (CK-D-12 3단계). |
| **L4** | 강제 완비 | 기준 동결 기계 강제 (CK-D-09) + 입구·출구 게이트 (spec 승인 + stop gate) + 하네스 자체 골든 테스트 (CK-C-07) + fail 방향 전수 선언 (CK-D-08). |
| **L5** | 적대적 성숙 | holdout 정보 비대칭 (CK-D-12 4단계) + 관측성 4종 완비 (CK-D-11) + 적대적 셀프 리뷰가 회귀 스위트에 편입 (IM-07-2) + 게이트 오탐률 데이터 기반 fail 방향 승격 이력 (AR-05-4). |

경험칙: 대부분의 실용 시스템의 목표는 **L4**다. L5는 검증 실패의 비용이 큰
시스템(자동 머지, 무인 배포)에서만 투자 대비 효익이 난다. L3 미만에서 자율
루프를 돌리는 것은 PR-03 매트릭스의 오른쪽 아래 사분면에서 에이전트를 돌리는 것과 같다.

## 부록 A — 최소 구현 스켈레톤 (동작 코드)

§3.3의 패턴을 실제 실행 가능한 최소 형태로 옮긴 것. POSIX sh 기준이며,
도구 호출 입력의 구체 스키마(JSON 필드명, exit code 의미)만 호스트 런타임
규약에 맞춰 바꾸면 된다. IM-10 1–2단계와 5단계의 출발점으로 그대로 쓸 수 있다.

**harness/gates/lib.sh** — 공용 파싱 단일 소스 (IM-03):

```sh
#!/bin/sh
# 모든 게이트가 source하는 공용 lib. 파싱 로직은 여기에만 존재한다.

STATE=".harness/state.md"

# field <file> <key> : "key: value" 라인에서 value 추출. 없으면 exit 1.
field() {
  v=$(grep -m1 "^$2:" "$1" 2>/dev/null | cut -d: -f2- | sed 's/^ *//; s/ *#.*//')
  [ -n "$v" ] && printf '%s\n' "$v"
}

# allow / deny <사유> : exit code 규약의 단일 정의 지점.
# 호스트 규약이 다르면 이 두 함수만 고친다.
allow() { exit 0; }
deny()  { printf 'BLOCKED: %s\n' "$1" >&2; exit 2; }

# gate_log <게이트명> <판정> <사유> : 발화 로그 append (AR-05-4)
gate_log() {
  run=$(field "$STATE" run_id) || run=none
  mkdir -p ".harness/runs/$run"
  printf '%s\t%s\t%s\n' "$1" "$2" "$3" >> ".harness/runs/$run/gate.log"
}
```

**harness/gates/stop_gate.sh** — 패턴 B의 완전판:

```sh
#!/bin/sh
# fail 방향: fail-open (흐름 제어용 — 상태 부재·필드 부재는 통과) [PR-04]
. "$(dirname "$0")/lib.sh"

phase=$(field "$STATE" phase)        || allow   # 필드 없음 → open
[ "$phase" = "done" ]                || allow   # done 선언이 아니면 무관

run=$(field "$STATE" run_id)         || deny "done인데 run_id 없음"
verdict=".harness/runs/$run/verdict.md"

[ -f "$verdict" ] || { gate_log stop deny no-verdict
                       deny "done 선언 전에 checker의 verdict가 필요"; }
[ "$verdict" -ot "$STATE" ] && { gate_log stop deny stale
                       deny "verdict가 마지막 상태 변경보다 오래됨 — 재채점"; }
grep -q ': FAIL' "$verdict" && { gate_log stop deny failing
                       deny "미통과 기준 존재: $(grep ': FAIL' "$verdict" | head -3)"; }
gate_log stop allow verified
allow
```

**harness/tests/run.sh** — 골든 테스트 러너 (IM-07):

```sh
#!/bin/sh
# 각 게이트에 fixture 상태를 주입하고 기대 exit code를 단언한다.
GATES=$(cd "$(dirname "$0")/../gates" && pwd)
pass=0; fail=0

expect() {  # expect <기대exit> <설명> <명령...>
  want=$1; desc=$2; shift 2
  "$@" >/dev/null 2>&1; got=$?
  if [ "$got" -eq "$want" ]; then pass=$((pass+1))
  else fail=$((fail+1)); echo "FAIL: $desc (want $want, got $got)"; fi
}

setup() {  # 임시 프로젝트 트리에 fixture 상태 구성
  T=$(mktemp -d); cd "$T" && mkdir -p .harness/runs/r1
  printf 'phase: %s\nrun_id: r1\n' "$1" > .harness/state.md
}

setup implement
expect 0 "done 아님 → open" sh "$GATES/stop_gate.sh"

setup done
expect 2 "done + verdict 없음 → block" sh "$GATES/stop_gate.sh"

setup done
printf -- '- R1: PASS — exit 0\n' > .harness/runs/r1/verdict.md
expect 0 "done + 전건 통과 → allow" sh "$GATES/stop_gate.sh"

setup done
printf -- '- R1: FAIL — exit 1\n' > .harness/runs/r1/verdict.md
expect 2 "done + FAIL 존재 → block" sh "$GATES/stop_gate.sh"

setup done
printf -- '- R1: PASS\n' > .harness/runs/r1/verdict.md
# 주의: mtime은 초 단위 해상도 — 같은 초의 touch로는 stale이 안 만들어진다.
touch -t 202001010000 .harness/runs/r1/verdict.md   # verdict를 과거로 강제
expect 2 "stale verdict → block" sh "$GATES/stop_gate.sh"

echo "$pass passed, $fail failed"; [ "$fail" -eq 0 ]
```

이 스켈레톤이 보여주는 규율의 실체:
- exit 규약·파싱·로그가 lib 한 곳에 (CK-C-01).
- fail 방향이 파일 머리에 선언되고, 골든 테스트가 오류 경로(verdict 없음, stale)를
  커버한다 (CK-C-02, CK-C-07).
- deny 메시지가 다음 행동을 알려준다 (CK-C-03).
- 테스트 러너 자체가 마지막 줄에서 비0으로 실패한다 — 침묵 실패 없음 (IM-08-2).
