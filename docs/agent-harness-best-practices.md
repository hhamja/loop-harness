# 에이전트 하니스 베스트 프랙티스

자율 코딩 에이전트 하니스를 설계·운영하기 위한 실무 레퍼런스.
Anthropic, OpenAI, 커뮤니티 리서치를 종합했다. 에이전트가 읽는 것을 전제로 작성됨: 규칙은 명령형이다 — 태스크가 명시적으로 덮어쓰지 않는 한 따를 것.

## 핵심 원칙

1. **컨텍스트 윈도는 제약이고, 디스크 위의 구조화된 산출물이 해법이다.** 아래 모든 항목은 세션 사이를 잇기 위해 존재한다.
2. **생성과 평가를 분리하라.** 에이전트는 자기 작업을 객관적으로 채점할 수 없다.
3. **세션당 태스크 하나.** 컨텍스트 고갈을 막고 세션을 복구 가능하게 유지한다.
4. **만들기 전에 검증하라.** 이전 세션이 무언가를 망가뜨렸을 수 있다고 가정할 것.
5. **빠른 피드백 루프를 배선하라.** 테스트, 린터, 타입 체커, UI 자동화가 backpressure 역할을 한다.
6. **모든 루프에 경계를 두라.** 반복 횟수 상한, 예산 한도, 기계 검증 가능한 정지 조건, 사람 에스컬레이션 경로.
7. **저장소가 단일 진실 원천이다.** 저장소에 없는 지식은 에이전트에게 존재하지 않는 것이다.
8. **지침은 권고이고, 훅은 보장이다.** 매번 반드시 일어나야 하는 것은 지침 파일이 아니라 훅에 둘 것.
9. **사람은 조종하고, 에이전트는 실행한다.** 엔지니어는 환경을 설계하고 의도를 명세하고 피드백 루프를 만든다; 코드는 에이전트가 쓴다.
10. **가차없이 단순화하라.** 하니스의 모든 구성요소는 "모델이 못 하는 것"에 대한 가정을 담고 있다; 그 가정은 모델 업그레이드마다 유효기간이 끝난다.

## 1. 아키텍처

- 단일 에이전트 루프에서 시작하라. 멀티 에이전트 오케스트레이션은 비결정성까지 겹친 마이크로서비스급 복잡도를 더한다; 단일 에이전트가 넘지 못하는 구체적 한계에 부딪힌 뒤에만 도입할 것.
- 분리하게 되면 관심사를 나눠라:
  - **Planner** — 짧은 프롬프트를 제품/기능 스펙으로 확장. 상위 수준 설계만; 초반에 과도하게 명세하면 오류가 하류로 연쇄된다.
  - **Generator** — 스펙에 맞춰 한 번에 기능 하나씩 구현.
  - **Evaluator** — 실행 중인 앱을 사용자처럼 테스트하고, 기준에 따라 채점하고, 구체적 피드백을 반환. 독립 evaluator를 회의적으로 튜닝하는 것이 generator를 자기비판적으로 만드는 것보다 훨씬 다루기 쉽다.
- compaction만 쓰기보다 세션마다 새 컨텍스트(컨텍스트 리셋)를 선호하라: 완전한 리셋은 "컨텍스트 불안"(컨텍스트가 차오르면 조기에 마무리하려는 경향)을 제거하고, 핸드오프 파일이 그 공백을 메운다. 모델이 발전하면 재평가할 것 — 유능한 모델에게는 compaction으로 충분해질 수 있다.

## 2. 상태와 영속성

세션을 넘어 살아남아야 하는 모든 것을 디스크에 영속화하라:

- **태스크/기능 목록** — Markdown이 아니라 JSON(모델이 일으키는 손상에 강함). 항목을 제거하거나 순서를 바꾸지 말 것; 상태만 미완료 → 완료로 뒤집을 것.
- **진행 노트** — 한 일, 발견/수정한 버그, 다음 할 일, 핵심 결정. 매 세션 끝에 작성.
- **계획/스펙 파일** — 원래 요구사항, 프로젝트 디렉터리에 보관.
- **init 스크립트** — 환경 셋업을 자동화해 설치에 컨텍스트를 낭비하지 않게 함.
- **git 히스토리** — 서술적인 커밋이 복구 메커니즘이다; 세션 시작 시 최근 커밋을 읽을 것.

최상위 지침 파일(AGENTS.md / CLAUDE.md)은 백과사전이 아니라 지도로 유지하라(~100줄): `docs/`를 가리키는 목차. 지식을 점진적 공개(progressive disclosure)로 구조화할 것 — 어디를 봐야 하는지 알려주는 작고 안정적인 진입점. 가장 중요한 지침을 앞에 둬서 잘려도(truncation) 사라지지 않게 할 것.

관련 지식을 전부 저장소로 밀어 넣어라 — 채팅 스레드의 결정, 아키텍처 패턴, 제품 컨텍스트. 에이전트가 발견할 수 없으면 읽을 수 없는 지식이다.

## 3. 세션 프로토콜

모든 세션을 이 순서로 진행하라:

1. **방향 잡기** — 진행 노트, 태스크 목록, 최근 git 히스토리 읽기
2. **셋업** — init 스크립트 실행
3. **베이스라인 검증** — 새 것을 건드리기 전에 기존 기능이 여전히 동작하는지 확인
4. **태스크 하나 선택** — 최우선순위 미완료 항목
5. **구현**
6. **테스트** — 유닛 테스트만이 아니라 실제 UI/API를 통해
7. **상태 갱신** — 태스크 완료 표시, 서술적 메시지로 커밋, 진행 노트 작성
8. **깨끗한 종료** — 애플리케이션을 동작하는 상태로 남길 것

세션당 태스크 하나. 프로젝트가 안정성을 입증한 뒤에만 완화하고, 품질이 떨어지면 다시 조일 것. 세션을 넘나들며 버그가 누적되는 것이 가장 흔한 실패 모드다 — 항상 베이스라인부터 검증하라.

## 4. 피드백 루프와 Backpressure

- 잘못된 출력을 거부할 수 있는 모든 것을 루프에 배선하라: 타입 체커, 린터, 테스트 스위트, 정적 분석기, 보안 스캐너. 피드백 바퀴는 빨리 돌아야 한다 — 느린 검증은 에이전트가 시도할 수 있는 반복 횟수를 깎아먹는다.
- 코드 단위를 구현한 직후 해당 테스트를 실행하라.
- 실행 중인 애플리케이션과의 상호작용을 강제하라(Playwright/Puppeteer MCP: 탐색, 클릭, 폼 입력, 스크린샷). 이것 없이는 에이전트가 테스트도 안 하고 기능을 완료로 표시한다.
- evaluator에게 "이거 괜찮아?"가 아니라 채점 가능한 기준을 줘라: good의 정의, 캘리브레이션용 few-shot 채점 예시, 확실한 탈락 임계값. 기준의 가중치는 모델이 이미 잘하는 것이 아니라 모델의 약점(디자인 독창성, 기능 완결성)에 두라.
- 복잡한 덩어리에는 **스프린트 계약**을 쓰라: generator가 무엇을 만들고 성공을 어떻게 검증할지 제안하고, evaluator가 구현 시작 전에 그 제안을 검토한다.
- evaluator는 태스크가 generator의 단독 역량 언저리이거나 그 너머일 때 값어치를 한다; 역량 한참 안쪽이면 evaluator는 오버헤드다. 그에 맞게 조정하라.

## 5. 컨텍스트 관리

- 주 컨텍스트는 스케줄러다. 검색, 코드 분석, 테스트 실행, 요약은 서브에이전트에 넘기고, 원본 덤프가 아니라 요약만 반환하게 하라.
- 읽기 전용 서브에이전트(검색, 분석)는 높은 병렬성으로 fan-out하고, 쓰기(빌드, 테스트)는 병렬성을 제한하라.
- 매 루프마다 같은 핵심 파일(계획, 스펙)을 결정적으로 로드해 매 반복이 같은 토대에서 시작하게 하라.
- 코드가 없다고 가정하지 말 것 — 에이전트 검색은 신뢰할 수 없다. 중복 구현을 피하려면 구현 전에 검색하라.

## 6. 프롬프트 규칙

- placeholder/stub 구현을 명시적으로 금지하고 완전한 구현을 강제하라 — 모델은 컴파일만 되는 최소 코드 쪽으로 치우쳐 있다. 필요하면 강한 표현을 쓸 것.
- 테스트와 아키텍처 결정의 "왜"를 문서나 주석에 남겨라 — 미래의 루프에는 원래 추론이 컨텍스트에 없다.
- 에이전트가 빌드/테스트/실행에서 배운 것을 AGENTS.md / CLAUDE.md에 반영하게 하라. 미래의 루프가 같은 실수를 반복하지 않도록.
- 발견한 버그는 현재 태스크와 무관해도 즉시 계획/todo 파일에 기록하라; 지금 고치거나 나중 루프에 남겨라.

## 7. 보안과 샌드박싱

심층 방어, 세 겹:

1. **OS 수준 샌드박스** — 실행 환경 격리
2. **파일시스템 제한** — 파일 작업을 프로젝트 디렉터리로 한정
3. **명령 allowlist** — 진짜 셸 렉서(예: `shlex`)로 파싱하고, 파이프/체이닝을 처리하고, 명시적으로 허용되지 않은 것은 전부 차단하고, 민감한 명령을 검증할 것(`pkill`은 dev 프로세스만, `chmod`는 `+x`만)

## 8. 코드 품질과 에이전트 가독성(legibility)

- 불변식은 문서만이 아니라 기계적으로 강제하라 — 린터, 구조 테스트, CI 체크. 린트 에러 메시지에는 시정 방법을 담아 에이전트가 사람 개입 없이 자가 수정할 수 있게 할 것.
- 규칙을 코드로 인코딩하라: 에이전트가 생성하는 코드베이스에서 인코딩된 규칙은 배수 효과를 낸다.
- 기술 부채를 가비지 컬렉션처럼 다뤄라: 반복 실행되는 정리 에이전트가 골든 원칙에서 벗어난 부분을 스캔해 작고 표적화된 리팩터링 PR을 연다.
- 지루한 기술(boring technology)을 선호하라 — 조합 가능하고 안정적인 API, 학습 데이터에 잘 표현된 스택 — 에이전트가 잘 모델링하지 못하는 최신 스택보다.
- 애플리케이션을 들여다볼 수 있게 만들어라: git worktree별 부팅으로 격리 인스턴스, DOM 스냅샷·스크린샷용 DevTools 프로토콜 배선, 로그/메트릭/트레이스를 조회 가능한 API로 노출.

## 9. 복구와 정지 조건

루프 시작 전에 정의하라 — 무한 루프는 하니스에서 가장 비싼 실패 모드다:

- **반복 횟수 상한** — 세션/시도 횟수의 확실한 최대치
- **예산 한도** — 세션당·실행당 토큰 또는 달러 상한
- **기계 검증 가능한 성공** — "done"은 통과하는 체크(테스트 green, exit 0, 파일 존재)이지, 에이전트 자신의 판단이 절대 아니다
- **포기 + 에스컬레이션** — 한 태스크에서 진전 없는 세션 N회 연속 → 멈추고 사람에게 에스컬레이션

git이 안전망이다: 태스크 성공마다 커밋, 정상 상태에 태그, 세션 시작 시 히스토리 읽기, 코드베이스가 망가지면 `git reset --hard` 후 재실행. 리셋-후-재실행과 프롬프트 구조 모두 유효하다 — 더 싼 쪽을 골라라.

계획을 주기적으로 재생성하라: 에이전트가 코드베이스를 스펙과 diff해 todo 목록을 다시 만들게 할 것. 계획은 표류하고, 낡은 계획은 오도한다.

최종적 일관성(eventual consistency)을 기대하라: 대부분의 문제는 루프를 더 돌리고 프롬프트를 더 다듬으면 해결된다.

## 10. 하니스 진화시키기

- 모델 업그레이드마다: 더 이상 하중을 받지 않는 스캐폴딩을 걷어내고, 새로 가능해진 역량을 위한 구성요소를 추가하고, 구성요소를 하나씩 제거해보며 영향을 검토해 테스트하라.
- 단순하게 시작하라; 복잡도는 구체적으로 입증된 한계에서만 추가하라. 비슷한 코드 세 줄이 섣부른 추상화보다 낫다.

## 도구별 노트

### Claude Code
- **CLAUDE.md**: 코드에서 유추할 수 없는 것만. 모든 줄에 대해 "이 줄을 지우면 실수가 생기나?"를 물을 것 — 아니라면 잘라내라. 가끔 필요한 지식은 **Skills**로(온디맨드 로드).
- 워크플로: **Explore → Plan → Code → Commit**; diff가 한 문장에 들어가면 계획은 생략.
- 검증 옵션, 싼 것부터 강한 것 순: 프롬프트 안의 명시적 테스트 케이스, 스크린샷 비교, `/goal` 조건, 결정적 게이트로서의 Stop 훅, 적대적 리뷰 서브에이전트. 성공했다는 주장이 아니라 증거를 요구하라.
- 스케일: CI와 fan-out에는 `claude -p`(headless), 병렬 세션에는 git worktree, 세션 간 writer/reviewer 분리.

### Codex
- **AGENTS.md 레이어링**: `~/.codex/AGENTS.md`(전역) → 저장소 루트 → 하위 디렉터리 오버라이드(가까운 쪽 우선); 기본 상한 32 KiB. 반복되는 임시 프롬프트 규칙은 즉시 AGENTS.md로 승격하라.
- 포함: 작업 합의, 빌드/테스트 명령, done 기준, 승인 절차. 제외: 긴 설명과 스택 설정(→ `config.toml`).
- 프롬프트 구조: **Goal / Context / Constraints / Done-when**.
- 안티패턴: 빌드 명령 미문서화, 복잡한 작업에서 계획 생략, 이해 전 전체 권한 부여, worktree 밖 작업, 프로젝트 전체를 하나의 거대 태스크로, 프롬프트에 방치된 낡은 임시 규칙.
